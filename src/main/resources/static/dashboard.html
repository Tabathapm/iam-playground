Fundamental

IAM (Identity & Access Management)

Disciplina que se encarga de gestionar identidades digitales y los accesos que esas identidades tienen sobre los sistemas. Responde dos preguntas centrales: ¿quién sos? (autenticación) y ¿qué podés hacer? (autorización).

Combina autenticación, autorización, gestión del ciclo de vida del usuario y auditoría. Es el "portero" digital de las empresas.

Fundamental

Autenticación (AuthN)

Probar la identidad de alguien. Ejemplos: usuario + contraseña, biometría, certificado digital, JWT, MFA. Responde a ¿quién sos?

Fundamental

Autorización (AuthZ)

Definir y verificar los permisos que tiene una identidad sobre los recursos. Ejemplos: "este usuario puede leer pero no borrar", "este rol puede acceder al panel admin". Responde a ¿qué podés hacer?

En HTTP, autenticación fallida = 401 Unauthorized; autorización fallida = 403 Forbidden.

Fundamental

SSO (Single Sign-On)

Login único: te logueás una vez y accedés a múltiples aplicaciones sin tener que volver a ingresar credenciales. Lo experimentás todos los días cuando entrás a Gmail y ya estás logueada en Drive, YouTube, etc.

Fundamental

Federación

Permitir que un usuario de una organización A acceda a sistemas de una organización B sin crear una cuenta nueva. La identidad de A es "confiada" por B.

Fundamental

RBAC (Role-Based Access Control)

Modelo de autorización donde los permisos se asignan a roles, y los usuarios reciben roles. En lugar de decir "Tabatha puede X, Juan puede Y", decís "los ADMINISTRADORES pueden X" y le das ese rol a quien corresponda. Escala mucho mejor que asignar permisos individualmente.

Fundamental

ABAC (Attribute-Based Access Control)

Modelo más granular que RBAC: los permisos se evalúan según atributos del usuario, recurso o contexto. Ejemplos: "puede acceder solo en horario laboral", "puede aprobar gastos hasta $5000", "puede acceder solo desde la red corporativa".

Fundamental

Claim

Afirmación verificable sobre un sujeto, en formato clave-valor. Por ejemplo: "sub": "tabatha", "roles": ["ADMIN"], "exp": 1234567890.

Los claims viajan dentro de tokens (JWT, ID Token de OIDC, SAML assertion). Son la unidad básica de información de identidad en sistemas modernos.

Fundamental

Provisioning / De-provisioning

Crear (provisioning) o quitar (de-provisioning) cuentas y accesos. Es el onboarding y offboarding de empleados en una empresa. Cuando alguien renuncia, hay que revocar todos sus accesos rápidamente.

Fase 1 — Scaffolding inicial

✅ Completada
Concepto

Spring Boot vs Spring Framework

Spring Framework (viejo): potente pero requiere mucha configuración XML manual. Spring Boot: trae auto-configuración + "starters" (paquetes que agrupan dependencias) + un servidor web embebido. Te ahorra horas de setup.

Ejemplo: con spring-boot-starter-web ya tenés Tomcat embebido, Jackson para JSON, validación, todo configurado.

Patrón

Arquitectura en capas (Controller → Service → Repository)

Separación clásica de responsabilidades:

Controller → Service → Repository → Base de datos

Controller: recibe HTTP, valida input, llama al service.
Service: lógica de negocio.
Repository: acceso a datos.

Anotación

Anotaciones REST de Spring

@RestController: marca la clase como controlador REST (devuelve JSON, no vistas).
@RequestMapping("/api"): prefijo común para todos los endpoints.
@GetMapping, @PostMapping: mapea método HTTP + ruta.
@PathVariable: captura variables de la URL (/api/usuarios/{id}).
@RequestBody: parsea el body JSON al objeto Java.

Concepto

Inyección de dependencias por constructor

Patrón donde las dependencias de una clase se reciben en su constructor, en lugar de crearlas adentro. Con Lombok @RequiredArgsConstructor, Spring inyecta automáticamente los servicios necesarios. Hace el código más testeable y desacoplado.

Buena práctica

DTO (Data Transfer Object)

Objeto que viaja entre capas. Separa lo que la API expone hacia afuera de lo que tenés internamente en la entidad. Esencial para nunca exponer cosas sensibles como el hash de la contraseña.

Fase 2 — JWT + PostgreSQL + BCrypt

✅ Completada
Concepto

JWT (JSON Web Token)

Formato de token auto-contenido y firmado. Tiene tres partes separadas por puntos: header.payload.signature.

Header (alg, typ) . Payload (claims) . Signature

El servidor firma con una clave secreta. Cualquiera puede leer el JWT (no está cifrado), pero solo el servidor puede generar uno válido. Si modifican el payload, la firma deja de coincidir y se rechaza.

Concepto

Claims registrados vs privados

Registrados (definidos por el RFC): sub (subject), iat (issued at), exp (expiration), iss (issuer), aud (audience), jti (id único).

Privados: los que define cada aplicación según sus necesidades. En la app se ven los claims roles e idUsuario dentro del payload del JWT.

Algoritmo

HS256, HS512, RS256

HS256/HS384/HS512: HMAC con SHA simétrico. La misma clave secreta firma y verifica. Bueno cuando solo tu servidor firma y verifica.

RS256: RSA asimétrico. Clave privada firma, clave pública verifica. Útil cuando hay varios servicios que verifican tokens emitidos por un servicio central (escenario típico de OIDC).

En la app se usa HS512 porque la clave secreta tiene 78 caracteres, longitud suficiente para ese algoritmo.

Concepto

Autenticación stateless

El servidor no guarda sesiones en memoria. La "prueba de identidad" viaja en el JWT que el cliente manda en cada request (Authorization: Bearer xxx). Permite escalar horizontalmente: cualquier instancia del servidor puede validar el token sin compartir estado.

Contraste con las sesiones tradicionales: el servidor guarda en memoria que "el usuario X está logueado" y el cliente manda una cookie con un id de sesión. La app evita esto: cada request lleva el JWT en el header Authorization.

Seguridad

Hashing vs Cifrado

Hashing: función unidireccional. No se puede des-hashear. Solo comparás hash contra hash. Se usa para contraseñas.

Cifrado: función bidireccional. Se puede descifrar con la clave correcta. Se usa para datos que necesitás recuperar en texto plano (ej. tarjetas de crédito).

Si un servicio ofrece "enviar la contraseña olvidada por email", es una mala señal: significa que la guarda en texto plano o cifrada, no hasheada.

Seguridad

BCrypt + Salt

BCrypt: algoritmo estándar de hashing de contraseñas. Incluye salt automático y costo configurable (factor de trabajo).

Salt: valor aleatorio único por contraseña. Hace que dos usuarios con la misma contraseña tengan hashes distintos. Protege contra ataques de rainbow tables.

En la app, los hashes guardados empiezan con $2a$10$... → versión de BCrypt + costo 10 + salt + hash. Se pueden ver en la columna hash_contrasena de la tabla usuarios.

Spring Security

Filter Chain de Spring Security

Spring Security funciona como una cadena de filtros HTTP. Cada request pasa por la cadena, y cada filtro puede autenticar, autorizar o rechazar.

En la app, FiltroAutenticacionJWT se inserta antes de UsernamePasswordAuthenticationFilter. Ese filtro extrae el JWT del header, lo valida, y setea el contexto de seguridad.

Spring Security

UserDetails + UserDetailsService

Spring Security tiene su propio modelo de "usuario" (UserDetails). Como la entidad Usuario de la app no implementa esa interfaz, se usa un adaptador (AdaptadorUsuarioSpring) que la envuelve.

El UserDetailsService es la pieza que Spring usa para "cargar" un usuario por username durante el proceso de autenticación.

JPA

JPA / Hibernate / Spring Data

JPA: specification de Java para mapear objetos a tablas (ORM).
Hibernate: la implementación más usada de JPA.
Spring Data JPA: capa sobre Hibernate que te genera repositorios automáticamente.

Con solo definir findByNombreUsuario(String) en la interfaz, Spring Data construye la query SQL automáticamente.

JPA

@Entity, @ElementCollection, @Enumerated

@Entity: marca la clase como mapeable a tabla.
@ElementCollection: para colecciones de tipos básicos o enums (se mapean a una tabla aparte; en la app es usuarios_roles).
@Enumerated(STRING): guarda el nombre del enum como texto, no como número (más legible y robusto a reordenamientos).

Validación

Bean Validation (@NotBlank, @Valid)

Estándar Java para validar inputs declarativamente. @NotBlank en un DTO + @Valid en el parámetro del controller = si llega vacío, Spring devuelve 400 antes de entrar al método. Patrón "fail fast".

Docker

Docker Compose + Volúmenes

Docker Compose: herramienta para definir y levantar múltiples contenedores con un YAML. Ideal para infraestructura local de desarrollo.

Volúmenes: persisten los datos del contenedor entre reinicios. Sin volumen, cuando hacés docker compose down se pierde la BD.

Fase 3 — RBAC con @PreAuthorize

✅ Completada
Spring Security

@EnableMethodSecurity

Anotación que "enciende" la posibilidad de usar @PreAuthorize y @PostAuthorize en métodos. Sin ella, esas anotaciones se ignoran silenciosamente, y eso es uno de los bugs más comunes de Spring Security.

Spring Security

@PreAuthorize

Evalúa una expresión SpEL (Spring Expression Language) antes de ejecutar el método. Si la expresión devuelve false, Spring devuelve 403 Forbidden y el método ni se ejecuta.

Ejemplos:
@PreAuthorize("hasRole('ADMINISTRADOR')")
@PreAuthorize("hasAnyRole('USUARIO', 'ADMINISTRADOR')")
@PreAuthorize("hasAuthority('ROLE_ADMIN')")

Convención

El prefijo "ROLE_"

Convención de Spring Security: al usar hasRole('ADMINISTRADOR'), Spring agrega automáticamente el prefijo ROLE_. Por eso en AdaptadorUsuarioSpring las authorities se guardan como "ROLE_" + rol.name().

Si usás hasAuthority(...) tenés que poner el prefijo explícito. Misma convención, dos formas de escribirla.

Concepto

401 vs 403 — autenticación vs autorización

401 Unauthorized: no sé quién sos (sin token, token inválido, token expirado). El nombre del status es históricamente confuso porque "unauthorized" suena a autorización, pero en realidad es de autenticación.

403 Forbidden: sé quién sos, pero no podés acceder a este recurso. Tu rol no tiene permisos suficientes.

Esta distinción es fundamental en cualquier sistema de IAM.

Principio

Separación de funciones

Principio de seguridad: ningún rol único debería tener todo el poder. Por eso en la app, ADMINISTRADOR no puede acceder a /api/auditor/logs aunque sea el "rol más alto". El AUDITOR puede leer logs que ni siquiera el admin debería poder alterar.

Fase 4 — OpenLDAP

✅ Completada
Concepto

Servicio de directorio

Base de datos especializada en almacenar información de identidades organizada jerárquicamente. Optimizada para lecturas masivas (typical 90% lecturas, 10% escrituras). Es la "agenda corporativa" digital.

Ejemplos: Active Directory (Microsoft), OpenLDAP (open source), Novell eDirectory (OpenText/NetIQ), Entra ID (ex Azure AD, la versión cloud de Microsoft).

Protocolo

LDAP (Lightweight Directory Access Protocol)

Protocolo estándar para consultar y modificar servicios de directorio. Definido en RFCs. Es a los directorios lo que SQL es a las bases relacionales.

Funciona sobre TCP/IP, puerto 389 (sin TLS) o 636 (con TLS, llamado LDAPS).

Terminología

DIT, DN, RDN

DIT (Directory Information Tree): el árbol entero del directorio.
DN (Distinguished Name): la "ruta única" de una entrada. Ej: uid=harry-ldap,ou=personas,dc=tabathapm,dc=local.
RDN (Relative Distinguished Name): el primer componente del DN (en el ejemplo, uid=harry-ldap).

Los DN se leen de izquierda (más específico) a derecha (más general). Es como una ruta de archivo invertida.

Atributos

Componentes de un DN

dc (domain component): partes del dominio. tabathapm.local → dc=tabathapm,dc=local.
ou (organizational unit): contenedor lógico (personas, grupos, computadoras).
uid (user id): identificador del usuario (el "username").
cn (common name): nombre para mostrar.
sn (surname): apellido (obligatorio en personas).
mail: email.
userPassword: contraseña hasheada.
memberOf: grupos a los que pertenece.

Concepto

objectClass

Cada entrada LDAP debe declarar a qué "clases" pertenece. La objectClass define qué atributos puede (o debe) tener la entrada. Ej: inetOrgPerson es la objectClass estándar para representar personas en una organización.

Operaciones

Operaciones LDAP

bind: autenticarse contra el directorio.
search: buscar entradas con filtros.
add: agregar entrada nueva.
modify: modificar atributos.
delete: borrar entrada.
unbind: cerrar la conexión.

Autenticación

Bind authentication (autenticación contra LDAP)

El patrón estándar de "login contra LDAP":

1. Bind como admin → 2. Buscar usuario por uid → 3. Bind con DN + password → ✅ o ❌

Hacen falta dos binds porque el usuario solo conoce su uid (ej. "harry-ldap"), no su DN completo. El primer bind sirve para encontrar el DN, el segundo para validar la contraseña.

Buena práctica

Service account / Usuario técnico

En IAM real, tu aplicación se conecta al directorio como una "cuenta técnica" con permisos limitados, NO como un usuario humano. En producción, esas credenciales se guardan en un vault (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault).

En la app se dejan en application.properties por ser un entorno de aprendizaje, aunque conocer esta práctica es importante: en producción irían en un vault.

Formato

LDIF (LDAP Data Interchange Format)

Formato de texto plano para importar/exportar datos en LDAP. Es el equivalente al INSERT de SQL para directorios. Cada bloque separado por línea en blanco es una entrada con su DN y atributos.

Para cargar usuarios al directorio se usa docker cp ldap-seed.ldif + ldapadd. Hay que tener en cuenta que docker compose down -v borra los volúmenes y deja el directorio vacío, así que después hay que reimportar el LDIF.

Implementado

Spring LDAP — LdapTemplate

Librería oficial para hablar con servidores LDAP desde Spring. LdapTemplate es similar a JdbcTemplate: abstrae las operaciones LDAP de bajo nivel.

Método clave: ldapTemplate.authenticate(base, filtro, contrasena) hace los dos binds (admin + usuario) automáticamente. El filtro va con paréntesis: (uid=tabatha-ldap).

Implementado

ConfiguracionLDAP (lo que armaste)

Spring no crea el LdapTemplate solo: hay que definirlo en una clase @Configuration. En la app se definen dos beans: un LdapContextSource (URL + credenciales del admin + base DN) y el LdapTemplate que lo usa.

Como el LdapContextSource ya tiene dc=tabathapm,dc=local como base, en la búsqueda solo pasás ou=personas — si repetís el base DN, se duplica y da el error "No Such Object" (code 32).

Implementado

Sincronización LDAP → PostgreSQL

Cuando un usuario LDAP se autentica por primera vez, tu ServicioAutenticacionLDAP lo crea automáticamente en PostgreSQL con rol USUARIO. Esto se llama just-in-time provisioning: la cuenta local se crea en el momento del primer login.

Resultado: la tabla usuarios queda con 6 registros — 3 locales (Fase 2) y 3 provenientes de LDAP (Fase 4). El usuario final obtiene el mismo JWT sin importar la fuente.

Logro

Dos fuentes de identidad, un mismo JWT

La app autentica contra dos lugares distintos: POST /api/auth/login (PostgreSQL + BCrypt) y POST /api/auth/login-ldap (OpenLDAP). Ambos devuelven el mismo tipo de JWT.

Esto es exactamente lo que hace un IAM real: ser una capa de abstracción que esconde dónde viven las identidades. En la app se puede ver al comparar las respuestas de /api/auth/login y /api/auth/login-ldap: distintos orígenes, mismo formato de token.

Fase 5 — OAuth 2.0 / OIDC

✅ Completada
Concepto

OAuth 2.0

Framework de autorización. Permite a una app acceder a recursos en nombre de un usuario, sin recibir su contraseña.

4 actores: Resource Owner (el usuario), Client (la app), Authorization Server (emite tokens), Resource Server (guarda los recursos).

Tokens: Access Token (vida corta, para llamar APIs), Refresh Token (vida larga, para renovar), Authorization Code (intermediario en flujos web).

Concepto

OIDC (OpenID Connect)

Capa de autenticación encima de OAuth 2.0. Es lo que se usa cuando hacés "Login with Google". Agrega el concepto de ID Token: un JWT con información de identidad del usuario.

OAuth te dice "esta app tiene permiso para acceder a X". OIDC además te dice "y este es el usuario".

Concepto

Flujos OAuth (grant types)

Authorization Code: el clásico para web apps con backend (el que usa la app).
Authorization Code + PKCE: para SPAs y mobile (evita exposición del client secret).
Client Credentials: máquina a máquina.
Device Code: para smart TVs y dispositivos sin teclado.

Concepto

Scopes

Permisos granulares en OAuth. Ej: profile, email, calendar.read, drive.write. La app pide los scopes openid email profile al iniciar sesión con Google.

Implementado

El flujo Authorization Code, paso a paso

Es el flujo que implementa la app: el usuario hace click en "Continuar con Google" → la app redirige al servidor de Google → el usuario autoriza → Google redirige a /api/auth/google/callback con un código → el backend intercambia ese código por tokens → extrae el email → genera un JWT propio.

El código es de un solo uso y de vida corta. La contraseña del usuario nunca pasa por la app: esa es la idea central de OAuth.

Implementado

ServicioAutenticacionGoogle

Hace un POST al token endpoint de Google (oauth2.googleapis.com/token) con el código, el client_id y el client_secret, usando RestTemplate. Google responde con un access_token y un id_token.

El intercambio usa application/x-www-form-urlencoded, no JSON: es un detalle del estándar OAuth fácil de pasar por alto.

Implementado

Decodificar el ID Token

El id_token es un JWT firmado por Google. La app decodifica su payload (la parte del medio, en Base64 URL-safe) para extraer el claim email y crear o ubicar al usuario en PostgreSQL.

Nota de seguridad: la app decodifica sin validar la firma porque controla el flujo de punta a punta. En producción se validaría la firma del id_token con la librería oficial de Google para evitar tokens falsificados.

Implementado

Frontend estático (login + panel)

Servido por Spring desde resources/static, en el mismo puerto que la API (sin CORS ni segundo servidor). login.html ofrece los tres métodos de login; dashboard.html muestra este resumen con barra de sesión (nombre, roles, cerrar sesión).

El JWT se guarda en localStorage y se decodifica en el cliente solo para mostrar el nombre y los roles. La protección real la hacen los endpoints de la API, no el HTML.

Logro

Tres fuentes de identidad, un mismo JWT

Con la Fase 5 la app autentica contra tres orígenes: PostgreSQL (local), OpenLDAP (directorio) y Google (OAuth/OIDC). Los tres terminan en el mismo JWT y pasan por la misma autorización por roles.

Esto es el corazón de un IAM: una capa única de identidad sobre fuentes heterogéneas. En la app se ve en login.html, con los tres botones que llevan al mismo panel.

Estándar

SAML 2.0

Estándar más viejo que OIDC, basado en XML. Cumple un rol similar a OIDC pero con otra implementación. Aún muy usado en empresas grandes y SSO corporativo (Microsoft, IBM, Oracle).

Concepto clave: SAML assertion, el equivalente al ID Token de OIDC.

Fase 6 · en curso

Tests (unitarios + integración)

Fase en curso. Con JUnit 5 + Mockito se cubren tres niveles: tests unitarios (lógica pura como ServicioJWT), tests de integración (los endpoints de login con MockMvc), y opcionalmente E2E (flujo completo login → JWT → recurso protegido).

El criterio: primero funcionalidad, después cobertura. El primer servicio cubierto es ServicioJWT, con 8 casos de prueba.

Framework

JUnit 5

Es el framework estándar para escribir tests en Java. Aporta las anotaciones (@Test, @BeforeEach) y los métodos de aserción (assertEquals, assertThrows).

Un test es, simplemente, código que prueba tu código. En vez de verificar a mano en Postman, lo automatizás: se vuelve una red de seguridad que avisa al instante si un cambio rompe algo.

Patrón

Patrón Arrange-Act-Assert (AAA)

Todo buen test sigue tres pasos, siempre en este orden. Arrange: preparar los datos. Act: ejecutar lo único que se quiere probar. Assert: verificar que el resultado sea el esperado.

Si la afirmación es falsa, el test falla. Mantener esta estructura hace los tests legibles y fáciles de mantener.

Anotación

@BeforeEach

Marca un método que corre antes de cada test. Sirve para que cada prueba arranque con un estado limpio y fresco, sin que una contamine a la otra.

En el test de ServicioJWT, el método setUp() crea un servicio y un usuario nuevos antes de cada uno de los 8 tests.

Anotación

@Test y @DisplayName

@Test marca un método como una prueba que JUnit debe ejecutar. @DisplayName es opcional: le pone una descripción legible al test, que aparece en el reporte en vez del nombre del método.

Es pura comodidad para leer qué pasó cuando se corren muchos tests juntos.

Aserciones

Las aserciones (assert)

Son el corazón del test. assertEquals(esperado, obtenido) falla si no coinciden. assertTrue / assertFalse para condiciones booleanas.

La más interesante en seguridad es assertThrows: verifica que el código lance una excepción. Por ejemplo, que un token firmado con otra clave sea rechazado — si no explotara, significaría que se aceptó un token falso.

Concepto

Failure vs Error

Dos resultados que se confunden. Un Failure es cuando un assert no se cumplió: el código anda, pero no como esperabas. Un Error es cuando el test reventó por algo inesperado (un NullPointerException, una clase que no existe).

Los failures apuntan a la lógica; los errors, muchas veces, a un problema de setup.

Buena práctica

Diseño testeable: inyección por constructor

ServicioJWT recibe sus dependencias por el constructor, así que para testearlo se instancia directo, sin levantar Spring ni mockear nada.

Esto no es casualidad: la inyección por constructor hace el código fácil de testear. Es un argumento de diseño sólido para defender en una entrevista.

Fase 7

Auditoría de seguridad

Registro de eventos de seguridad para análisis posterior: quién se logueó, cuándo, desde dónde, qué intentó hacer. Esencial para compliance (GDPR, SOX, HIPAA).

En la Fase 7 se guardarán logs de auditoría en la base de datos (login exitoso, login fallido, cambio de roles, etc.).

Pregunta de repaso

"¿Qué entendés por IAM?"

Respuesta: "IAM es la disciplina que se encarga de gestionar identidades digitales y los accesos que esas identidades tienen sobre los sistemas. Combina autenticación, autorización, gestión del ciclo de vida del usuario y auditoría."

Pregunta de repaso

"¿Cuál es la diferencia entre autenticación y autorización?"

Respuesta: "Autenticación es probar quién sos — usuario y contraseña, un JWT válido, un certificado. Autorización es definir qué podés hacer una vez autenticado. En HTTP: 401 es fallo de autenticación, 403 es fallo de autorización. En la app, el JWT autentica al usuario, y las anotaciones @PreAuthorize aplican autorización por rol."

Pregunta de repaso

"¿Conocés LDAP?"

Respuesta: "Sí. LDAP es el protocolo estándar para consultar servicios de directorio como Active Directory, OpenLDAP o eDirectory. La información se organiza jerárquicamente con DNs y un DIT. Las operaciones principales son bind (autenticación), search, modify. En la app se integró OpenLDAP con Spring Boot — los usuarios pueden autenticarse contra el directorio o contra una base de datos local, y obtienen el mismo JWT."

Pregunta de repaso

"¿Qué es OAuth?"

Respuesta: "OAuth 2.0 es un framework de autorización que permite a aplicaciones acceder a recursos en nombre del usuario sin tener su contraseña. OIDC es la capa de autenticación encima de OAuth — lo que usás cuando hacés Login with Google. El flujo más común es Authorization Code, donde el cliente recibe un código que intercambia por un access token."

Pregunta de repaso

"¿Cómo funciona JWT?"

Respuesta: "Un JWT tiene tres partes: header con el algoritmo, payload con los claims (información del usuario), y signature. El servidor firma con una clave secreta. Cualquiera puede leer el contenido pero solo el servidor con la clave puede generar uno válido. En la app se usa HS512, los claims incluyen sub, roles, iat, exp. La autenticación es stateless: el server no guarda sesiones, todo viaja en el token."

Pregunta de repaso

"¿Por qué BCrypt y no SHA256?"

Respuesta: "BCrypt está diseñado específicamente para contraseñas: incluye salt automático, es lento a propósito (factor de costo configurable) lo que dificulta ataques por fuerza bruta. SHA256 es rápido — algo bueno para checksums pero malo para contraseñas, porque un atacante puede probar millones de combinaciones por segundo. La regla es: hashes generales rápidos para datos, hashes lentos para contraseñas."

Pregunta de repaso

"¿Qué ventaja tiene soportar varias fuentes de identidad?"

Respuesta: "Permite que usuarios de orígenes distintos —una base local, un directorio LDAP corporativo, o un proveedor como Google— accedan al mismo sistema sin duplicar lógica. La aplicación queda como una capa de abstracción sobre la identidad. En la app se ve con los endpoints /api/auth/login y /api/auth/login-ldap, que validan contra fuentes diferentes pero devuelven el mismo JWT."

Pregunta de repaso

"¿Cómo funciona Spring Security?"

Respuesta: "Spring Security usa una cadena de filtros HTTP. Cada request pasa por múltiples filtros que pueden autenticar, autorizar o rechazar. En la app, un filtro personalizado para JWT se ejecuta antes de los filtros por defecto, extrae el token del header Authorization, lo valida, y setea el contexto de seguridad con la información del usuario. Después, anotaciones como @PreAuthorize evalúan la autorización."
